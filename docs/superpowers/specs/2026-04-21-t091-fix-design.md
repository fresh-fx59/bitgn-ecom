# t091 fix — case-insensitive search + collection-giveup nudge

**Status:** proposal, awaiting local validation.
**Source failure:** `artifacts/ws_snapshots/t091_ocr_badger/run_0/analysis.md`.
**Scope:** one file (`src/bitgn_contest_agent/adapter/pcm_tracing.py`) + one nudge section in `src/bitgn_contest_agent/verify.py`.

## Context

t091 ("Work the oldest inbox message" → "OCR all bills related to Badger") is the one failure in 103/104. Root cause is two-layered:

1. **PCM `search` is case-sensitive.** Agent searched `badger`; bill files contain `Badger`. Three zero-hit searches convinced the agent no evidence exists.
2. **Agent treats `search` as authoritative for negative evidence.** Never fell back to `list`+`read` on `50_finance/purchases/`.

Verify fired `INBOX_GIVEUP` correctly but the nudge is tuned for "find sender" tasks, not collection-over-entity tasks.

## Proposed fixes

### Fix A — client-side case-fold wrapper

**File:** `src/bitgn_contest_agent/adapter/pcm_tracing.py:173` (the `search` method).

**Change:** when an LLM-issued `search` returns 0 hits AND the pattern is a single alphanumeric token (no regex metachars) that is all-lowercase or all-uppercase, automatically retry with the title-cased form and merge hits. Transparent to the LLM; appears as a single `search` op in the trace.

**Why this shape:**
- No PCM server change (out of our control).
- Catches t091 and any other case-mismatch silently across all 104 tasks.
- Single-token proper-noun queries rarely false-positive under case-fold (a full case-insensitive grep would; a targeted title-case retry on empty results does not).
- Skipping the retry when the pattern already has mixed case preserves agent intent (`"Badger"` already tried → don't retry `"badger"`).

**Pseudocode:**
```python
def search(self, req):
    result = self._inner.search(req)
    hits = result.results  # protobuf field
    if len(hits) == 0 and _is_single_lowercase_token(req.pattern):
        retry_req = _with_title_case(req)
        retry_result = self._inner.search(retry_req)
        if retry_result.results:
            # merge, preserve original pattern in trace observability
            result = _merged(result, retry_result)
    return result
```

**Test vectors:**
- `search badger` on content `Badger` → hits via retry
- `search BADGER` → skips retry (not lowercase); relies on original match
- `search "purchases/.*\\.md"` → skips retry (has regex metachars)
- `search "project X"` → skips retry (not a single token)
- `search Badger` → primary hit returns; no retry fired

### Fix B — INBOX_GIVEUP nudge extension for collection tasks

**File:** `src/bitgn_contest_agent/verify.py:220-239` (`_section_inbox_giveup`).

**Change:** when the inbox task text contains a collection quantifier (`all`, `every`, `each`) AND an entity name, add to the nudge: *"Before concluding no evidence, `list` the lane most likely to hold these records (`50_finance/purchases/` for bills, `30_knowledge/notes/` for notes, etc.) and `read` each candidate directly. Do not rely on `search` for negative evidence — a case-sensitive miss is not proof of absence."*

**Why this shape:**
- Fix A helps silently when the agent *does* search. Fix B covers the path where:
  - The bill/note uses an alias that differs from the task word
  - The entity is referenced by relationship token, not name
  - The structural linkage is directory-based (not content-searchable)
- The `all`/`every`/`each` trigger is a cheap, deterministic gate — no LLM call.

**Pseudocode:**
```python
def _section_inbox_giveup(task_text: str) -> str:
    base = (...existing copy...)
    if re.search(r"\b(all|every|each)\b", task_text, re.IGNORECASE):
        base += (
            "\n  - This inbox item names a collection. Before concluding "
            "no evidence exists, `list` the lane most likely to hold these "
            "records and `read` each candidate — do not rely on `search` "
            "for negative evidence on collections, as case-sensitive misses "
            "and alias mismatches can hide matches."
        )
    return base
```
